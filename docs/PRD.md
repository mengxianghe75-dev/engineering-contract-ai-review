# 工程合同 AI 审查助手 — 产品需求文档（PRD）

## 一、产品概述

### 1.1 产品名称

工程合同 AI 审查助手

### 1.2 产品定位

面向工程合同场景的智能审查与管理平台，支持多角色协作、可配置规则引擎、文档管理与审查版本留痕。产品定位为**可交付给客户的原型系统**，同时支持内部教学演示与后续商业化迭代。

### 1.3 目标用户

| 角色 | 职责 | 典型场景 |
|------|------|----------|
| 管理员（admin） | 系统配置、用户管理、规则配置、全局查看 | 配置大模型接入、管理团队成员、维护审查规则库 |
| 审查员（reviewer） | 合同上传、触发审查、编辑合同信息 | 日常合同审查工作流 |
| 只读用户（viewer） | 查看合同与审查结果、导出报告 | 了解项目合同状态，无需操作 |

### 1.4 核心价值

- **规则优先 + AI 补充**的混合审查策略，在无大模型时也能完整跑通
- 后台可配置规则与大模型参数，**不需要改代码**即可调整
- 支持同一合同多版本上传与差异对比，**满足工程合同频繁修订**的实际需求
- 全操作留痕可审计，**满足合规与汇报需求**

## 二、功能需求

### 2.1 登录与权限

#### 2.1.1 功能描述

- 用户名/密码 JWT 登录
- 三级角色体系：admin / reviewer / viewer
- 菜单、页面、接口三层权限控制

#### 2.1.2 权限矩阵

| 功能 | admin | reviewer | viewer |
|------|-------|----------|--------|
| 查看合同列表 | ✓ | ✓ | ✓ |
| 上传合同 | ✓ | ✓ | — |
| 编辑合同信息 | ✓ | ✓ | — |
| 触发审查 | ✓ | ✓ | — |
| 导出报告 | ✓ | ✓ | ✓ |
| 用户管理 | ✓ | — | — |
| 规则管理 | ✓ | — | — |
| 系统设置 | ✓ | — | — |
| 操作日志 | ✓ | — | — |

#### 2.1.3 默认账号

系统首次启动自动初始化管理员账号：

| 用户名 | 密码 | 角色 |
|--------|------|------|
| admin | Admin123456 | admin |

### 2.2 合同管理

#### 2.2.1 功能描述

- PDF 上传，支持单文件最大 50MB
- 文本型 PDF 直接提取文本内容
- 扫描型 PDF 通过 OCR（PyMuPDF + Tesseract）识别
- 大页数扫描件走后台异步队列
- 合同列表支持多条件筛选
- 合同详情查看，包含基础信息、审查结果、版本列表

#### 2.2.2 合同属性

| 字段 | 说明 | 是否可编辑 |
|------|------|------------|
| 合同名称 | 自动从 PDF 文件名提取 | 是 |
| 项目编号 | 用户手动输入 | 是 |
| 归属用户 | 上传者默认 | 是 |
| 分类 | 自由输入（如"施工合同"） | 是 |
| 标签 | 自由输入，多标签 | 是 |
| 状态 | uploaded / parsed / reviewed / archived | 是 |
| 归档时间 | 归档时自动记录 | 否 |

#### 2.2.3 筛选条件

合同列表支持以下筛选：

- 合同名称（模糊搜索）
- 归属用户
- 分类
- 标签
- 风险等级
- 创建时间范围
- 包含/排除归档合同

### 2.3 审查引擎

#### 2.3.1 功能描述

审查流程包含三个阶段：

1. **规则引擎匹配** — 按优先级执行配置的审查规则
2. **内置启发式补充** — 系统内置的通用工程合同风险规则
3. **LLM 风险识别** — 可选，通过 OpenAI 兼容 API 补充额外风险

#### 2.3.2 基础字段提取

自动从合同文本中提取以下字段：

- 合同名称
- 合同编号
- 合同类型（施工合同 / 分包合同 / 采购合同等）
- 签约日期
- 合同金额
- 甲方 / 乙方
- 工期 / 开工日期 / 竣工日期
- 付款条款
- 质保金条款

#### 2.3.3 风险识别

每条风险项包含：

| 字段 | 说明 |
|------|------|
| code | 风险编码（如 payment_terms_risk） |
| title | 风险名称 |
| level | 风险等级（high / medium / low） |
| matched_text | 命中原文片段 |
| description | 风险说明 |
| recommendation | 建议 |
| source | 来源（rule / builtin / ai） |
| rule_id | 关联规则 ID（规则命中时） |

#### 2.3.4 审查模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| mock | 模拟模式，不依赖外部模型 | 产品演示、开发测试 |
| openai_compatible | OpenAI 兼容模式 | 接入真实大模型 |

真实模型调用失败时，自动降级为 mock 模式，provider 记录为 mock_fallback。

#### 2.3.5 审查摘要

每次审查生成一段中文摘要，包含整体风险概括和重点提示，支持以下两种生成方式：

- **mock 摘要** — 基于风险项自动生成的模板摘要
- **LLM 摘要** — 由大模型生成的专业摘要

### 2.4 规则管理

#### 2.4.1 功能描述

管理员可在后台配置审查规则，规则按优先级排序执行。

#### 2.4.2 规则字段

| 字段 | 说明 |
|------|------|
| 规则名称 | 如"付款条款风险" |
| 规则编码 | 唯一标识，如 payment_terms_risk |
| 风险类型说明 | 描述该规则检测的风险类型 |
| 条件类型 | keyword / regex / contains_any / contains_all |
| 条件值 | 关键词、正则表达式或 JSON 数组 |
| 风险等级 | high / medium / low |
| 建议 | 命中后给出的建议 |
| 优先级 | 数值越小越先执行 |
| 启用状态 | 启用 / 停用 |
| 合同类型范围 | 留空表示全部，可限定特定合同类型 |

#### 2.4.3 条件类型

| 类型 | 说明 | 示例 |
|------|------|------|
| keyword | 关键词匹配 | "预付款" |
| regex | 正则表达式 | `\d{4}年\d{1,2}月` |
| contains_any | 命中任一关键词 | `["甲方", "乙方"]` |
| contains_all | 命中全部关键词 | `["甲方", "乙方"]` |

### 2.5 版本管理

#### 2.5.1 功能描述

同一业务合同支持多次上传 PDF，形成版本链。支持两个维度的版本管理：

- **上传版本** — 同一合同组的多个文件版本（V1 / V2 / V3）
- **审查版本** — 单个文件多次审查的历史快照

#### 2.5.2 版本对比

支持对比同一合同组下两个上传版本，差异覆盖：

- 基础字段变化
- 新增风险
- 移除风险
- 风险等级变化
- 运行模式变化（模型降级/恢复记录）

#### 2.5.3 版本导出

支持导出指定上传版本的 PDF 审查报告。

### 2.6 操作日志

#### 2.6.1 功能描述

系统自动记录关键操作，仅管理员可查看。

#### 2.6.2 记录范围

| 操作类型 | 说明 |
|----------|------|
| 上传合同 | 新合同上传或新版本上传 |
| 编辑合同 | 修改合同属性 |
| 触发审查 | 发起合同审查 |
| LLM 链路降级 | 真实模型调用失败降级 |
| LLM 链路恢复 | 模型恢复正常 |
| 导出报告 | PDF 报告导出 |
| 新增/编辑规则 | 规则配置变更 |
| 启用/停用规则 | 规则状态变更 |
| 新增/编辑用户 | 用户管理操作 |
| 启用/停用用户 | 用户状态变更 |

### 2.7 系统设置

#### 2.7.1 功能描述

管理员可在前端页面直接配置大模型参数，不需要修改代码或重启服务。

#### 2.7.2 可配置项

| 配置项 | 说明 |
|--------|------|
| 审查模式 | mock / openai_compatible |
| 大模型接口地址 | LLM_BASE_URL |
| 大模型 API Key | LLM_API_KEY |
| 大模型名称 | LLM_MODEL |
| 超时时间（秒） | 默认 30 |

#### 2.7.3 连通性测试

提供"测试连通性"按钮，发送最小请求验证大模型接口是否可连通。

### 2.8 报告导出

#### 2.8.1 功能描述

将审查结果导出为 PDF 格式的审查报告。

#### 2.8.2 报告内容

- 合同基础信息
- 审查状态与版本信息
- 审查摘要
- 风险项清单（含命中原文与建议）
- 运行模式说明

## 四、部署方案

### 4.1 Docker Compose 部署

```bash
docker compose up --build -d
```

包含三个服务：

| 服务 | 端口 | 说明 |
|------|------|------|
| postgres | 5432 | PostgreSQL 数据库 |
| backend | 8000 | FastAPI 后端 |
| frontend | 5173 | Nginx 静态服务 |

### 4.2 本地开发部署

```bash
# 1. 启动数据库
docker compose up -d postgres

# 2. 启动后端
cd backend
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload

# 3. 启动前端
cd frontend
npm install
npm run dev
```

### 4.3 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| DATABASE_URL | postgresql://... | 数据库连接 |
| JWT_SECRET_KEY | change-this... | JWT 签名密钥 |
| REVIEW_PROVIDER | mock | 审查模式 |
| LLM_BASE_URL | "" | 大模型接口地址 |
| LLM_API_KEY | "" | 大模型 API Key |
| LLM_MODEL | "" | 大模型名称 |
| CORS_ORIGINS | "*" | 跨域白名单 |

## 六、非功能需求

| 指标 | 目标 |
|------|------|
| 合同上传响应 | < 5s（< 10MB） |

- 浏览器：Chrome / Edge / Safari 最新版本
- Python：3.9+
- 数据库：PostgreSQL 14+
- Docker：20.10+

## 八、关键业务流程

### 8.1 合同审查流程

```
上传 PDF
    ↓
提取文本（文本型直接提取 / 扫描型走 OCR）
    ↓
提取结构化字段
    ↓
规则引擎匹配（按优先级）
    ↓
内置启发式规则补充
    ↓
LLM 风险补充（如启用）
    ↓
合并结果（规则优先，去重）
    ↓
生成审查摘要
    ↓
保存审查版本
    ↓
生成审查报告（可导出）
```

### 8.2 版本对比流程

```
选择基准版本和目标版本
    ↓
加载两个版本的审查结果
    ↓
对比基础字段（新增 / 修改 / 删除）
    ↓
对比风险项（新增 / 移除 / 等级变化）
    ↓
对比运行模式（模型降级 / 恢复记录）
    ↓
生成差异报告
```

## 九、风险与约束

### 9.1 当前约束

- 当前阶段不提供合同删除入口，后续建议支持软删除
- 标签目前为自由输入，后续建议升级为词典
- OCR 依赖 Tesseract，大页数文档解析时间较长
- LLM 模式下需要配置第三方 API，存在网络依赖

### 9.2 已知风险

- 扫描件清晰度影响 OCR 准确率
- LLM 输出格式不合法时会被服务端丢弃
- 规则优先级设置不当可能导致误报
- 大模型 API Key 泄露风险（已做加密存储）
